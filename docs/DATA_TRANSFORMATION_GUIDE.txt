제어 로직 도면 데이터 구조화
- DCS 지역난방 계열 자동화를 위한 도면 전처리 작업 -

작성일: 2026-02-05


■ 배경

지역난방 발전소의 DCS(분산제어시스템)를 효율적으로 운영하고
향후 자동화 및 AI 지원 기능을 구현하기 위해서는
제어 로직 도면을 컴퓨터가 이해할 수 있는 형태로 변환하는 작업이 선행되어야 함.
이 문서는 그 전처리 작업에 대한 내용임.


■ 한 일

발전소에는 수백 장의 제어 로직 도면이 PDF 파일로 존재함.
기존에는 로직을 확인하려면 PDF를 직접 열어서 눈으로 연결선을 따라가야 했음.

이 문제를 해결하기 위해 PDF 도면에서 블록과 연결선 정보를 자동으로 추출하여
컴퓨터가 읽을 수 있는 JSON 데이터로 변환함.
이제 태그를 검색하면 해당 태그가 어느 도면에 있는지 바로 찾을 수 있고,
어떤 블록과 연결되어 있는지도 확인 가능.


■ 대상 도면

1) 제어 로직 도면
   - 출처: Emerson Ovation에서 PDF로 출력한 도면
   - 내용: 밸브나 펌프가 어떤 조건에서 작동하는지를 나타낸 자동 제어 논리
   - 예시: "온도가 100도를 넘으면 밸브를 연다" 같은 조건들이 블록으로 표현됨

2) P&ID 도면
   - 출처: AutoCAD로 작성된 DWG 파일을 DXF로 변환
   - 내용: 밸브와 계기가 현장 어디에 위치하는지를 나타낸 배치도
   - 용도: 제어 로직에서 찾은 밸브가 실제로 어디 있는지 확인할 때 사용


■ 원본 PDF 조건 (중요)

이 방식이 가능한 이유는 Emerson Ovation에서 출력한 PDF가 "디지털 PDF"이기 때문임.

스캔한 PDF는 불가:
  종이 도면을 스캐너로 찍은 PDF는 그냥 사진이라서
  글자도 그림으로 인식되고, 블록 색상도 정확하지 않음.
  이런 파일은 데이터 추출이 어려움.

Ovation PDF의 특징:
  - 텍스트가 내장되어 있음: 블록 이름, 태그명이 글자 데이터로 들어있어서
    PyMuPDF로 바로 읽을 수 있음. OCR(글자 인식)이 필요 없음.
  - 벡터 그래픽: 블록이 색상이 있는 사각형으로 그려져 있음.
    빨간색은 OCB 블록, 녹색은 SIGNAL, 노란색은 ALG 블록.
  - 연결선: 블록 사이를 잇는 검은 선이 명확하게 그려져 있음.

PDF에 좌표 정보가 있나?
  PDF 자체에는 "이 블록의 좌표가 (120, 340)이다" 같은 정보가 없음.
  좌표는 PDF를 이미지로 변환한 후 OpenCV로 분석해서 획득.
  이미지에서 빨간색 영역을 찾으면 그 픽셀 위치가 블록의 좌표가 됨.


■ 변환 방법

제어 로직 (PDF → JSON):

  PDF 자체에는 "블록 A가 블록 B에 연결됨" 같은 구조화된 정보가 없음.
  따라서 이미지 분석을 통해 데이터 추출.

  1단계: PDF를 이미지로 변환
         pdf2image로 PDF 각 페이지를 PNG 이미지(300 DPI)로 변환.
         이미지 크기는 약 5450 x 3520 픽셀.

  2단계: 텍스트 추출
         PyMuPDF로 PDF에 내장된 텍스트 추출.
         블록 이름(OCB008802C), 태그명(FCV-1551), 포트명(IN, OUT) 등 획득.

  3단계: 블록 위치 검출 (좌표 추출)
         OpenCV로 이미지 분석.
         - 색상 필터링: HSV 색공간에서 빨간색/녹색/노란색 영역 분리
         - 외곽선 검출: 각 색상 영역의 사각형 경계 탐색
         - 결과: 블록의 픽셀 좌표 (x, y)와 크기 (width, height) 획득
         예: 빨간 블록이 픽셀 (120, 340) 위치에 80x60 크기로 존재

  4단계: 연결선 검출
         OpenCV의 HoughLinesP 함수로 검은 선 탐색.
         선의 시작점과 끝점 좌표를 얻어서 어떤 블록에서 어떤 블록으로
         이어지는지 파악.

  5단계: JSON으로 저장
         블록 정보(이름, 타입, 좌표)와 연결선 정보를 JSON 파일로 저장.


P&ID (DWG → JSON):

  P&ID는 CAD 파일이라서 이미 좌표 정보가 포함되어 있어 추출이 용이함.

  1단계: AutoCAD에서 DWG 파일을 DXF 형식으로 변환

  2단계: ezdxf 라이브러리로 DXF 파일을 읽어서
         밸브 위치, 계기 위치, 텍스트 정보를 추출

  3단계: JSON으로 저장


■ 추출 결과

제어 로직 도면: 223개 도면, 총 585페이지
태그 수: 5,979개 (하나의 태그가 여러 도면에 나올 수 있음)
  * 태그란 설비/신호의 고유 식별번호
    예: TIT2641(온도전송기), FCV-1551(유량조절밸브), PIT2621(압력전송기)

P&ID 도면: 55개
밸브: 3,118개
계기: 3,466개


■ JSON 데이터 예시

1) 제어 로직 블록 데이터

  하나의 블록(AND 게이트)이 다음과 같이 저장됨:

  {
    "AND_1145_2288": {
      "type": "BLOCK_TYPE",
      "cx": 1144.8,
      "cy": 2288.2,
      "ports": [
        { "name": "OUT", "type": "PORT", "cx": 1195.6, "cy": 2279.2 },
        { "name": "IN1", "type": "SIGNAL", "cx": 1067.3, "cy": 2239.2 },
        { "name": "IN2", "type": "SIGNAL", "cx": 1065.3, "cy": 2319.2 }
      ]
    }
  }

  원형 데이터 (PDF에서 추출):
  - 블록 이름(AND): PyMuPDF로 텍스트 추출
  - 좌표(cx, cy): OpenCV로 이미지 분석하여 픽셀 위치 추출
  - 포트 이름(IN1, OUT 등): PyMuPDF로 텍스트 추출

  변형 데이터 (활용을 위해 가공):
  - 블록 ID(AND_1145_2288): 블록명 + 좌표를 조합하여 고유 ID 생성
  - type 분류: 블록 색상과 텍스트를 분석하여 종류 분류
  - ports 배열: 블록 주변 텍스트를 포트로 그룹화


2) 포트 연결 정보 (핵심 데이터)

  어떤 포트에서 어떤 포트로 신호가 흐르는지 저장됨:

  {
    "fromName": "TIT2641_386_234",
    "fromCx": 386,
    "fromCy": 234.2,
    "toName": "NUM",
    "toParent": "OCB0088007",
    "toCx": 339.1,
    "toCy": 504,
    "waypoints": [
      { "x": 339.1, "y": 234.2 }
    ]
  }

  원형 데이터 (PDF에서 추출):
  - 좌표(fromCx, toCx 등): OpenCV HoughLinesP로 선 검출
  - waypoints: 연결선이 꺾이는 지점 좌표

  변형 데이터 (활용을 위해 가공):
  - fromName, toName: 좌표와 블록 위치를 매칭하여 연결 관계 생성
  - toParent: 도착 포트가 속한 블록 이름

  해석: "TIT2641의 신호가 OCB0088007 블록의 NUM 포트로 입력됨"

  이 연결 정보가 있어야 "이 신호가 어디로 가는가?" 추적이 가능함.
  PDF 원본에는 이런 연결 관계가 명시되어 있지 않고,
  좌표 분석을 통해 "이 선의 끝점이 어느 포트에 닿는가"를 계산해서 생성한 데이터임.


3) P&ID 밸브 데이터

  하나의 밸브가 다음과 같이 저장됨:

  {
    "id": "CV-0001",
    "type": "CHECK",
    "symbol": "check",
    "size": "50",
    "description": "INITIAL FILLING",
    "x": 1220.3,
    "y": 73.1,
    "grid": "5A",
    "rotation": 0,
    "layer": "V-NUMBER",
    "variants": ["CV-0001", "CV0001", "CV 0001"]
  }

  원형 데이터 (DXF에서 직접 추출):
  - id: 밸브 번호 (텍스트 레이어에서 추출)
  - x, y, rotation, layer: DXF 파일에 원래 있던 값
  - symbol: DXF 블록 이름 (예: "check", "gate", "globe")

  변형 데이터 (활용을 위해 가공):
  - type: symbol 값을 기반으로 밸브 종류 분류 (CHECK, GATE, GLOBE 등)
  - size: 주변 텍스트에서 사이즈 정보 추출
  - grid: x,y 좌표를 그리드 위치로 변환 (5A = 5열 A행)
  - variants: 검색 편의를 위한 밸브번호 변형 (하이픈 유무, 공백 등)
  - description: 주변 텍스트에서 추출한 용도 설명


■ 현재 활용

  - 태그 검색 기능 구현: 5,979개 태그 즉시 검색 및 도면 위치 확인
  - 연결 추적 기능 구현: 블록 간 신호 흐름 추적
  - P&ID 위치 확인: 밸브/계기 현장 그리드 좌표 표시


■ 향후 확장 가능성

  - 블록 연결 정보 텍스트 변환 → AI 전달 가능
  - 로직 변경 시 영향 범위 자동 분석
  - 운전원 질의응답 지원 ("이 로직을 바꾸려면?")
  - 알람 발생 시 관련 로직 빠른 추적

  → DCS 운영 자동화 및 AI 지원 시스템 기초 데이터 확보


■ 현재 한계

  - 제어 로직 구조(블록 연결 관계)만 파악 가능
  - 물리적 현상 진단 불가 ("왜 펌프에서 진동이 나는가?")
    → 기계적 원인(베어링 마모 등)은 로직 데이터로 분석 불가능

  향후 보완 방안:
  - 설비별 정비 매뉴얼 연동
  - 정비 이력 데이터 연동
  - 로직 구조 + 정비 데이터 결합

  → "이 알람 발생 시 과거 조치 사례" 조회 시스템으로 확장 가능


■ 사용 도구

  Claude Code(Anthropic) 활용 개발
  - 라이브러리 선정, 코드 작성, 디버깅, 문서 작성 전 과정 지원

  도입 도구:
  - PyMuPDF: PDF 텍스트 추출
  - pdf2image: PDF → 이미지 변환
  - OpenCV: 블록 검출, 연결선 추적
  - ezdxf: DXF 파싱
  - HTML5 Canvas + JavaScript: 뷰어 구현
